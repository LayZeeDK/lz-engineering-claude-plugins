## Summary

This phase is a near-mechanical mirror of the already-shipped Phases 7 (Fowler, 62 leaves) and 8 (Kerievsky, 27 leaves). Both the shipped catalog pattern (thin README index + one leaf per entry, filename == kebab-slug of the H1) and the non-shipped validation harness (`.claude/skills/lz-refactor-workspace/`, node-builtin checkers + a one-module-per-fence `tsc --strict` extractor) already exist and are GREEN today (verified: check-kerievsky 27/27, check-catalog 62/62, check-crossrefs 449 links resolve). The planner does NOT need to invent structure -- it clones `check-catalog.mjs` into `check-gof.mjs`, adds a small `check-extra-patterns.mjs`, retrofits one branch into `check-kerievsky.mjs`, and adds two entries each to `check-crossrefs.mjs` and `extract-samples.mjs`, then wires two commands into `package.json`. The single GoF-specific delta in the checkers is the leaf field set (5 GoF sections instead of the Fowler `Use when:`/Motivation/Mechanics fields) and the Applicability-first-line selector mirror (instead of the `Use when:` mirror).

The content is sourced through the identical clean-room oracle loop: author blind, run the deterministic layer first, gate fidelity + DST-04 near-verbatim via the isolated `oracle-reviewer` against `.oracle/design-patterns/` (GoF-23) and `.oracle/refactoring-to-patterns/` (the 5 extra), revise blind, cap ~3 rounds, escalate to the owner only on non-convergence. GoF is now fully provisioned in `.oracle/` (verified: index.md + Ch.3/4/5 + glossary), so -- unlike Phase 8 -- the loop gates GoF directly rather than via AskUserQuestion.

The instrument-first RED baseline is the load-bearing thing to get right in planning: after the Wave-1 harness plan lands, `check-gof` is 0/23, `check-extra-patterns` is 0/5, and `check-kerievsky` flips from 27/27 GREEN to RED on the ~23 un-linkified pattern tokens. All three are the EXPECTED Nyquist scaffold state, not regressions; they turn GREEN as Wave 2 authors leaves and Wave 3 linkifies + finalizes.

**Primary recommendation:** Clone-and-narrow, do not rebuild. `check-gof.mjs` = `check-catalog.mjs` with the field set swapped to the 5 GoF sections + Applicability-first-line mirror + a 3-entry required-Away map + a Singleton-cites-DI assertion. The linkify gate in `check-kerievsky.mjs` should be link-parse + set-membership driven off the GoF-23 (23) and extra (5) name sets -- this is the one correctness-critical piece, because the token `Factory` (extra pattern) must NOT be confused with `Factory Method` (GoF-23).

### Pattern: name-identity contract (reuse verbatim)

Every checker asserts each canonical name appears EXACTLY once as a leaf H1 (identity, not cardinality), filename == `slugFor(name)`, single H1, no unknown/typo heading, no draft-scaffolding phrases (`SCAFFOLD_RES`). `check-catalog.mjs` lines 152-327 are the template. `slugFor` (line 127) and `readmeRowsBySlug` (line 184-208) transfer unchanged. The README-row regex `/\]\(([a-z0-9][a-z0-9-]*)\.md\)/g` matches bare `](slug.md)` links (no anchor) -- so gof/extra README rows MUST link with a bare `.md` (no `#anchor`), matching the Fowler/Kerievsky index convention.

### Anti-Patterns to Avoid

- **Rebuilding the checker from scratch.** Clone `check-catalog.mjs`; only the field set + selector-mirror differ.
- **Substring-matching pattern names in the linkify gate.** `Factory` (extra) is a prefix of nothing but is a distinct token from `Factory Method` (GoF-23). Match on the extracted name against the exact name sets, never substring.
- **Filling gof/extra README rows in Wave 1.** Rows that link to not-yet-authored leaves would make `check-crossrefs` RED for the wrong reason. Wave-1 READMEs carry family headings + intro only; rows land in Wave 3 (D-05 "Wave 3 finalize").
- **Spawning gsd-executor for Wave-2 content plans.** It has no Agent/AskUserQuestion tools; the oracle loop fails structurally (Phase 7/8 documented failure). `autonomous: false`, main-context inline (D-13).
- **Reading `.oracle/` prose in the main context.** DST-04 firewall; `ls` only.

### S1b. `check-kerievsky.mjs` linkify branch (retrofit -- the correctness-critical piece)

**Replace with the linkify branch.** Recommended design = **link-parse + set-membership**, driven off the GoF-23 (23) and extra (5) name sets (duplicate the two small arrays into check-kerievsky, or a shared token map):

For each leaf, read the `GoF pattern:` line and:
- Parse the first Markdown link on the line, if any: `[<text>](../<catalog>/<slug>.md#<anchor>)`.
- Determine the declared pattern NAME: if a link is present, name = link text; else name = the leading token before ` (` or end of line.
- **If name is in the GoF-23 set:** REQUIRE a link to `../gof-catalog/<slugFor(name)>.md#<slugFor(name)>`. FAIL if absent or pointing to the wrong catalog/slug.
- **If name is in the extra-5 set:** REQUIRE a link to `../extra-patterns-catalog/<slugFor(name)>.md#<slugFor(name)>`. FAIL if absent/wrong.
- **If name is in NEITHER set** (the `n/a -- utility` sentinels + `Class / type-safe value`): REQUIRE that NO gof/extra link is present (free text). FAIL if a stray link appears.

This naturally produces the RED baseline: pre-Wave-3 the 23 pattern tokens are plain names in-set with no link -> 23 FAILs; the 4 free-text tokens have names not in either set -> PASS. Post-Wave-3 all 23 carry correct links -> PASS. RESOLUTION of the link target is left to `check-crossrefs` (division of labor). The critical correctness detail: `Factory` maps to `extra-patterns-catalog/factory.md`, NOT to `Factory Method` (`gof-catalog/factory-method.md`) -- the exact-name set lookup guarantees this; substring matching would break it.

## Common Pitfalls

### Pitfall 1: `Factory` vs `Factory Method` token confusion in the linkify gate
**What goes wrong:** A substring or prefix match sends the extra-pattern token `Factory` to `gof-catalog/factory-method.md`, or fails to distinguish them.
**Why:** GoF-23 has `Factory Method` (and `Abstract Factory`); the extra catalog has a distinct `Factory`. The tokens differ but overlap lexically.
**How to avoid:** Exact-name set lookup (GoF-23 set vs extra-5 set), never substring. Encoded in the S1b design.
**Warning signs:** check-crossrefs RED with `factory.md not found` or a Kerievsky Factory leaf linking to `factory-method.md`.

### Pitfall 2: `.oracle/` chapter truncation on the long Behavioral chapter
**What goes wrong:** `oracle`/`oracle-reviewer` silently drop the tail of a long chapter (Read ~25K-token cap), yielding false `source-only`/missed-drop verdicts. [CITED: 08-LEARNINGS.md]
**Why:** GoF Ch.5 Behavioral (11 patterns) and Ch.4 Structural (7 patterns) are large single chapters -- likely well over 25K tokens.
**How to avoid:** The agents were fixed to chunk-read to EOF (committed 9a877d7-era, reloaded); still, sub-batch the Behavioral chapter (D-15 splits it into 2 plans of ~6 and ~5) and run an incomplete-read sanity check on big-chapter verdicts. This is exactly why D-15 splits Behavioral.
**Warning signs:** A reviewer verdict claims a pattern is absent from the source, or flags many leaves `wrong` at once.

### Pitfall 3: Blind draft converges on GoF's canonical example
**What goes wrong:** The blind draft's `## Example` mirrors the book's well-known illustration (the document-editor/Lexi case study; maze/MapSite; the composite file-system) -> `too_close_to_source` DST-04 flag. [CITED: 08-LEARNINGS.md "blind drafts accidentally converge on the source example"]
**How to avoid:** Choose a deliberately distinct domain per leaf (D-04's example must name the participants + collaboration in original code). Re-domain on a `too_close` verdict.
**Warning signs:** oracle-reviewer `example` axis returns `too_close_to_source` with a structural reason.

### Pitfall 4: Filling gof/extra README rows during Wave 1
**What goes wrong:** Wave-1 README rows link to leaves that do not exist yet -> `check-crossrefs` RED for the wrong reason, muddying the intended RED baseline.
**How to avoid:** Wave-1 READMEs = family headings + intro/gloss only, NO leaf-linking rows. Rows land in Wave 3 (D-05 "Wave 3 finalize"). check-gof's Applicability-first-line mirror is vacuous at 0 leaves and stays RED-for-present-leaves through Wave 2 (expected).

### Pitfall 5: Reopening converged leaves on a lone reviewer re-flag
**What goes wrong:** A fresh adversarial reviewer pass re-flags an already-`pass` leaf, churning committed content. [CITED: 08-LEARNINGS.md oscillation]
**How to avoid:** Honor the ~3-round cap and the converged verdict; re-gate ONLY a changed axis. Escalate genuine non-convergence to the owner rather than looping.

### Pitfall 6: GSD phase-close auditor emits non-ASCII
**What goes wrong:** A phase-close auditor's generated artifact contains em dashes / curly quotes, tripping the ASCII-only constraint. [CITED: 08-LEARNINGS.md]
**How to avoid:** Re-scan generated phase-close artifacts (VERIFICATION/SECURITY/VALIDATION) for ASCII; `check-hygiene` hard-fails only on shipped `references/**`, but the repo constraint is ASCII-everywhere.
