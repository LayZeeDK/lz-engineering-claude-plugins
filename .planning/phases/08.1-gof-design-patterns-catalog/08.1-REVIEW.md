## Warnings

### WR-01: `extract-samples.mjs` silently drops an unterminated `ts` fence from compilation

**File:** `.claude/skills/lz-refactor-workspace/extract-samples.mjs:60-89`
**Issue:** `extractFences` only pushes a fence onto the result list when it sees the *closing*
` ``` ` toggle (the `else` branch at line 74-78). If a `ts` fence is opened but never closed
(EOF reached with `open !== null`), the fence is never emitted, never written to `samples/`,
and never type-checked -- with no warning. This harness is the sole executable gate over the
catalog's only executable surface, so a malformed fence would silently *reduce* compile
coverage rather than fail. The "213 modules GREEN" guarantee therefore has a silent hole: a
broken example hidden behind a missing close-fence would pass as green.
**Fix:** After the line loop, detect the dangling-open state and fail loudly, e.g.:
```js
  return fences;
};

// ... after extractFences is called per file, or inside it before return:
if (open !== null) {
  throw new Error(`unterminated code fence in ${file}`);
}
```
(Track the source file name into `extractFences`, or check `open` at the call site and abort
with a non-zero exit so an authoring mistake surfaces as RED instead of shrinking coverage.)

### WR-02: divergent slug algorithms across the checker suite (`slugFor` vs `slugsFor`)

**File:** `.claude/skills/lz-refactor-workspace/tools/check-kerievsky.mjs:161` and `.claude/skills/lz-refactor-workspace/tools/check-crossrefs.mjs:53-83`
**Issue:** Two different slug algorithms coexist in the reviewed checker suite:
- `slugFor` (in check-gof/check-extra/check-kerievsky) does `replace(/[^a-z0-9]+/g, "-")`, so
  a non-alphanumeric run becomes a hyphen.
- `slugsFor` (in check-crossrefs, modeling GitHub's anchor generation) does
  `replace(/[^\w\s-]/g, "")` then `replace(/\s+/g, "-")`, so punctuation is *dropped*, not
  hyphenated.
For a name containing a character GitHub drops (e.g. the `/` in the Kerievsky name
"Replace One/Many Distinctions with Composite"), `slugFor` yields `...one-many...` while
`slugsFor`/GitHub yield `...onemany...`. `check-kerievsky` builds its required link target's
anchor with `slugFor` (line 337/343), but `check-crossrefs` validates anchors with `slugsFor`
-- so for such a name the two checkers would demand contradictory anchors and one would always
fail (and the `slugFor`-style anchor would also be dead on github.com).
This is DORMANT for the in-scope catalogs: all 23 GoF and 5 extra names are alphanumeric +
spaces, for which the two functions agree exactly, so no in-scope link is affected and the
linkify gate is safe today. It is a latent trap for any future pattern name carrying droppable
punctuation.
**Fix:** Have both files derive slugs from a single shared function that matches GitHub's rule
(the `slugsFor` punctuation-dropping variant), so filename derivation, link-target derivation,
and anchor validation cannot diverge.

## Info

### IN-01: `ts` fence presence check is not scoped to the `## Example` section

**File:** `.claude/skills/lz-refactor-workspace/tools/check-gof.mjs:267,275` (same pattern in `check-extra-patterns.mjs:220,228`)
**Issue:** The "## Example present" check and the "has a `ts` fence" check are independent
regexes run over the whole leaf text. A leaf could satisfy both while its `## Example` section
is empty and the fence lives elsewhere. All current leaves place the fence inside Example, so
this is a coverage gap, not a live defect.
**Fix:** Extract the `## Example` section body (the existing `sectionBody` helper already does
this for Consequences) and assert the ` ```ts ` fence appears within it.

### IN-02: Decorator example's retry branch is never exercised by the demo

**File:** `plugins/lz-tdd/skills/lz-refactor/references/gof-catalog/decorator.md:56-65`
**Issue:** `RetryingHandler.handle` returns `first` when `first.length > 0`, else re-invokes
`this.inner.handle(request)`. The wrapped `CoreHandler` always returns `handled:${request}`
(never empty), so the fallback branch is dead in the demo and the decorator adds no
observable behavior to the run. The structure is a correct Concrete Decorator; only the
illustrative payoff is muted. Consider a decorator whose added responsibility is visibly
exercised (e.g. a logging/counting wrapper, or a core handler that can return empty).

### IN-03: `check-crossrefs.mjs` double-counts failures (cosmetic)

**File:** `.claude/skills/lz-refactor-workspace/tools/check-crossrefs.mjs:173-183,193`
**Issue:** Each unresolved link calls `report(false, ...)` inline (incrementing `failures`),
and then the aggregate `report(unresolved === 0, "all cross-links resolve", ...)` at line 193
adds one more failure. The final `failures` count in the RED summary is therefore inflated.
Pass/fail correctness is unaffected (any unresolved link still yields exit 1).
**Fix:** Make the per-link reports the source of truth and turn line 193 into a
non-failing informational log, or drop the inline per-link `failures++` and rely on the
aggregate.
