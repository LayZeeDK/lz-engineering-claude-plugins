## Phase Boundary

Phase 8.1 fills the seam Phase 8 left open. Phase 8 authored all 27 Kerievsky
pattern-directed refactorings, each carrying a `GoF pattern:` token as **dangling
free text** (no link target). Phase 9's coach assumes a pattern reference layer
exists to route toward and away from. Phase 8.1 stands up a GoF design-patterns
reference catalog (original prose + original TypeScript, clean-room-verified like
Fowler/Kerievsky) that (a) resolves the dangling `GoF pattern:` cross-refs into real
links, (b) gives the Phase 9 coach a concrete pattern target to name/justify toward
and away, and (c) solidifies canonical TypeScript idioms for the classic patterns,
shared across all three catalogs. It is a content phase shaped like Phases 7-8,
inserted as decimal **8.1** (QA is byte-identical to an integer phase). Satisfies
GOF-01..GOF-04 + XTR-01.

**In scope (the CONTENT of the GoF pattern layer):**

1. **GoF-23 catalog** (`references/gof-catalog/`) -- all 23 GoF patterns as
   per-pattern leaf files behind a thin README index, each on the locked 5-section
   contract (Intent / Applicability / Consequences / Example / Related patterns),
   original prose + `tsc --strict`-clean original TypeScript, grouped by family
   (Creational / Structural / Behavioral) in the README with the Applicability-first-
   line selector mirror. Author-cited modern-status caveats folded into Consequences.
2. **Extra-patterns catalog** (`references/extra-patterns-catalog/`) -- the 5 Tier-1
   extra patterns (Null Object, Factory, Creation Method, Composed Method, Collecting
   Parameter) as leaves on the same 5-section contract, Kerievsky-grounded.
3. **Kerievsky linkify retrofit** -- every Kerievsky `GoF pattern:` token that names
   a real pattern becomes a resolvable link (GoF-23 -> gof-catalog, extra -> extra-
   patterns-catalog); the 3 Direction:Away de-patterning links land; utility/type-code
   tokens stay free text.
4. **Harness extension** -- extend the non-shipped checker battery (new check-gof +
   check-extra-patterns, extended check-crossrefs / check-kerievsky / extract-samples /
   package.json) to gate the 23 + 5 leaves, their contracts, and the linkify branch.
5. **SKILL.md wiring** -- one lean pointer section to the two new catalog READMEs.

All TS samples are `tsc --strict`-clean; the GoF-23 content is verified via the
clean-room `oracle-reviewer` loop over `.oracle/design-patterns/`, the 5 extra
patterns over `.oracle/refactoring-to-patterns/` (the main context never reads book
prose -- DST-04).

**Out of scope (later phases / future milestone):**

  8.1 does NOT touch the Phase-9 coach placeholder; Phase 9 owns routing.
- Beck/Feathers principle-backing (Phase 9, PRIN-*); version bump / README / CHANGELOG

### Scope & Completeness

- **D-01:** Author ALL 23 GoF patterns as per-pattern leaves -- no curated subset. Grounded in the GoF authors' own modern view ("Design Patterns 15 Years Later", InformIT 2009; Artima 2005 Gamma interviews): they "still love them all"; the only draft-drop was Singleton (kept with caveat); their intended change was tiering/annotation + additions, never subtraction. A curated subset was explicitly rejected. Locked completeness set by family: Creational (5) Abstract Factory, Builder, Factory Method, Prototype, Singleton; Structural (7) Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy; Behavioral (11) Chain of Responsibility, Command, Interpreter, Iterator, Mediator, Memento, Observer, State, Strategy, Template Method, Visitor. [Wave 2]
- **D-02:** Author EXACTLY 5 Tier-1 extra patterns -- Null Object, Factory, Creation Method, Composed Method, Collecting Parameter -- non-GoF-23 patterns Kerievsky documents with pattern-grade material (oracle-verified sourceable from the 3 books). Tier-2 patterns (Type Object, Extension Object, a Dependency Injection leaf) are deferred to a future milestone (external sources only, not in the oracle books) and are OUT of this phase. [Wave 2]

### Catalog Structure & Layout

- **D-03:** Ship two sibling catalogs under `plugins/lz-tdd/skills/lz-refactor/references/`: `gof-catalog/` (the 23) and `extra-patterns-catalog/` (the 5). Each is a thin `README.md` index over per-pattern leaf files; each leaf filename = the kebab-case slug of its H1 pattern name. (Directory names are approved/locked.) [Wave 1 scaffolds READMEs; Wave 2 adds leaves]
- **D-04:** Every leaf uses the locked 5-section contract: `## Intent` (what it does + the problem it targets), `## Applicability` (positive situations + symptoms; FIRST LINE = the one-sentence index selector), `## Consequences` (benefits AND liabilities where the source states them; modern caveat cited here), `## Example` (ONE strict-TS-compiling fence whose identifiers/comments name the participants and wire their collaboration -- this absorbs GoF's Structure/Participants/Collaborations), `## Related patterns` (cross-links + distinctions). Optional leading `Also known as:` line. NO synthesized `Avoid when`, NO invented `Watch for`, NO one-line `Use when:` selector. The 5 extra-pattern leaves reuse this exact structure. [Wave 1 encodes in check-gof/check-extra-patterns; Wave 2 authors to it]
- **D-05:** The README index selector for each row = the leaf's `## Applicability` FIRST LINE, mirrored verbatim (enforce "Applicability first line = standalone one-sentence selector"; no separate `Use when:` label). README grouping is index-side (headings in the README) by the 3 GoF families (Creational / Structural / Behavioral), mirroring how the Fowler index groups by chapter. NO per-leaf `Category:` label (siblings carry no taxonomy token). [Wave 1 scaffold; Wave 3 finalize]

### Modern-Status Caveats & Consequences (GOF-02)

- **D-06:** Fold author-cited modern-status caveats INTO `## Consequences` (NOT a new section) for the flagged patterns: Singleton (design smell; prefer DI), Iterator (language-subsumed), Interpreter + Flyweight (niche / "different beasts"), Factory Method (Gamma would generalize to "Factory"), Composite/Template Method/Command (JUnit 3->4 meta-programming), Observer (often native). Singleton's Consequences MUST cite Dependency Injection (Fowler's DI/IoC article) as the preferred alternative. Sources: informit p=1404056; artima how-to-use-design-patterns, erich-gamma-on-flexibility-and-reuse. [Wave 2]
- **D-07:** Consequences rule -- section present + populated; require a LIABILITY only where the source states one. Do NOT mandate a liability for all 23: the 8 GoF-benefits-only patterns (Builder, Singleton, Bridge, Facade, Proxy, Command, Iterator, Template Method) carry benefits-only faithfully (+ modern caveat where the authors give one, e.g. Singleton); mandating a liability would force invention. Collecting Parameter's explicit liabilities list is empty -- assemble its downside from Kerievsky's surrounding prose + the Visitor contrast (tied to specific classes/interfaces; reads poorly across heterogeneous objects where Visitor is preferred); documented, NOT fabricated beyond what that prose supports. [Wave 1 encodes "present + populated, liability only where sourced"; Wave 2 authors]

### Cross-Linking & the Kerievsky Retrofit (GOF-03)

- **D-08:** Retrofit every Kerievsky `GoF pattern:` token into one of three outcomes: (a) a `gof-catalog` link when it names a GoF-23 pattern; (b) an `extra-patterns-catalog` link for the 5 extra patterns (`Introduce Null Object` -> Null Object; `Encapsulate Classes with Factory` + `Move Creation Knowledge to Factory` -> Factory; `Replace Constructors with Creation Methods` -> Creation Method; `Compose Method` -> Composed Method; `Move Accumulation to Collecting Parameter` -> Collecting Parameter); (c) free text ONLY for tokens naming no real pattern -- the 3 `n/a -- utility` sentinels (Chain Constructors, Unify Interfaces, Extract Parameter) and `Replace Type Code with Class` (a type-safe-class value, adjacent to the deferred Type Object). Links use the established `../<catalog>/<slug>.md#<slug>` H1-anchor convention (no sub-section anchors). [Wave 1 encodes the linkify gate in check-kerievsky; Wave 3 applies the links]
- **D-09:** Cross-links are bidirectional -- each gof leaf's `## Related patterns` links back to the Kerievsky refactorings that reach it. Require an `## Related patterns` link to the Kerievsky Direction:Away refactoring for EXACTLY 3 patterns: Singleton -> Inline Singleton; Composite -> Encapsulate Composite with Builder; Iterator -> Move Accumulation to Visitor. No other pattern requires an Away link. (Direction values are settled by the book's Refactoring Directions table per project memory `kerievsky-direction-table-authoritative`.) [Wave 1 encodes the required-3 rule; Wave 2/3 author]

### Harness / Checker Changes (all under `.claude/skills/lz-refactor-workspace/`, NON-shipped)

- **D-10:** Add new `tools/check-gof.mjs` (clone `check-catalog.mjs`): locked-23 `NAMES` grouped by family; leaf contract = single H1 + filename==slug + the 5 sections + optional `Also known as:` + Applicability-first-line mirrored into a README row + no scaffolding phrases + README grouped by the 3 families. Encodes the Consequences rule (present + populated; liability only where sourced; 8 benefits-only patterns exempt) and the Related->Away required-3 rule (Singleton/Composite/Iterator only). Add new `tools/check-extra-patterns.mjs` (small clone): locked NAMES = {Null Object, Factory, Creation Method, Composed Method, Collecting Parameter}; same 5-section contract; same Consequences rule. Assert name IDENTITY, not cardinality. [Wave 1]
- **D-11:** Extend the rest of the battery in the SAME Wave-1 harness plan (the one plan that owns ALL checker edits): `check-crossrefs.mjs` gains `gof-catalog` + `extra-patterns-catalog` in its source set so their outbound links resolve; `check-kerievsky.mjs` replaces the presence-only `GoF pattern:` check with the linkify branch (GoF-23 name MUST link to `../gof-catalog/<slug>.md#<slug>`; extra name MUST link to `../extra-patterns-catalog/<slug>.md#<slug>`; token naming no real pattern MUST NOT carry a gof/extra link); `extract-samples.mjs` gains `gof-catalog` + `extra-patterns-catalog` entries in its `CATALOGS` array (per-catalog prefix prevents sample collisions); `package.json` inserts `check-gof.mjs` + `check-extra-patterns.mjs` into the `check` chain after `check-kerievsky.mjs`. `check-hygiene.mjs` needs no change (already walks `references/**`). The RED baseline (0/23, 0/5, check-kerievsky RED on un-linkified tokens) is the EXPECTED instrument-first Wave-1 state, not a regression. [Wave 1]

### Oracle Model (clean-room, carried forward from Phases 7-8, LOCKED)

- **D-12:** Source and verify ALL gof-catalog + extra-pattern content via the clean-room `oracle`/`oracle-reviewer` loop (Model C), IDENTICAL to Phases 7-8: author BLIND from public knowledge; run the DETERMINISTIC layer first (tsc + contract + hygiene + cross-ref checks) before spending reviewer calls; gate fidelity + full-strength DST-04 near-verbatim via the isolated `oracle-reviewer` (its verdict never carries source prose); revise BLIND from the directives; sub-batch ~6-8 leaves per reviewer call in parallel; cap ~3 rounds; escalate non-convergence / `ambiguities` to the owner via AskUserQuestion. GoF-23 content is sourced/gated against `.oracle/design-patterns/` (index.md + Ch.3 Creational / Ch.4 Structural / Ch.5 Behavioral); the 5 extra patterns against `.oracle/refactoring-to-patterns/`. The main context NEVER reads `.oracle/` prose (only `ls` for names). Reference only `index.md` + chapter number/topic in `.planning/` artifacts -- never hardcode the book's chapter filenames. (GoF is NOW in `.oracle/`, so -- unlike Phase 8's D-04 AskUserQuestion fallback for GoF -- the oracle loop gates GoF directly; AskUserQuestion survives only for reviewer escalations.) [Wave 2]
- **D-13:** The orchestrator (MAIN context) drives the Wave-2 content plans INLINE -- NOT via `gsd-executor`. `gsd-executor` has no `Agent` tool (cannot spawn `oracle`/`oracle-reviewer`) and no `AskUserQuestion` (cannot do owner escalation), so only the main context can run the author -> gate -> revise -> escalate loop. Set `branching_strategy=none`, no worktrees (the oracle agents are read-only). This is the Phase-7/8 blocking anti-pattern carried forward -- the plan MUST encode `autonomous: false` on the oracle-loop content tasks or execution fails the same way. [Wave 2]
