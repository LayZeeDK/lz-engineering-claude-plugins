# Phase 08.1 Learnings: gof-design-patterns-catalog

## Decisions

### Wave-2 authoring runs INLINE in the main context (D-13)
The 5 clean-room authoring plans (02-06) ran inline in the orchestrator's own context, NOT via gsd-executor and NOT in worktrees.

**Rationale:** the loop needs the `Agent` tool (to spawn read-only `oracle`/`oracle-reviewer` subagents) and `AskUserQuestion` (owner escalation) -- a gsd-executor subagent has neither, so it fails structurally on the loop. The oracle agents are read-only, so there is no file-conflict reason to isolate. Wave 1 (harness) and Wave 3 (linkify/finalize) are deterministic and were delegated to gsd-executor.
**Source:** 08.1-CONTEXT.md (D-13), 08.1-RESEARCH.md, 08.1-02..06-PLAN.md

### Clean-room firewall: the main context never reads .oracle/
Book content was reached ONLY through the isolated oracle-reviewer subagent; the authoring context used `ls` for names only and authored blind.

**Rationale:** copyrighted GoF / Kerievsky expression must never cross into the public repo (DST-04). The reviewer holds the full text precisely because its verdict carries none of it back.
**Source:** 08.1-02..06-SUMMARY.md, 08.1-SECURITY.md (T-08.1-01)

### Paraphrase every canonical Intent one-liner (DST-04 hardening)
All 23 GoF Intent sentences were rewritten into original wording rather than reproduced near-verbatim.

**Rationale:** the GoF one-line Intents are distinctive definitional prose; reproducing them verbatim is avoidable shared expression on a public repo. Applied uniformly across all leaves once the pattern was identified.
**Source:** 08.1-02..05-SUMMARY.md (oracle re-gate directives)

### The deterministic checker battery IS the Nyquist test suite
No unit-test runner was added; the node-builtin exit-code checkers + `tsc --strict` extractor + `claude plugin validate .` are the validation instruments.

**Rationale:** the catalog is Markdown whose only executable surface is one strict-TS fence per leaf; the battery samples every requirement (GOF-01..04, XTR-01) with a GREEN automated signal. Fabricating unit tests would be redundant.
**Source:** 08.1-VALIDATION.md

### Factory (extra) H1 kept exactly "Factory" to disambiguate from Factory Method
The Kerievsky "Factory" leaf's H1 is exactly `# Factory` so the Wave-3 set-membership linkify resolves the two Factory tokens into extra-patterns-catalog/factory.md, never gof-catalog/factory-method.md.

**Rationale:** exact-name (never substring) set membership is the only safe disambiguation between the extra Factory and GoF-23 Factory Method (Pitfall 1).
**Source:** 08.1-06-SUMMARY.md, 08.1-01-PLAN.md

### .worktreeinclude added to carry .oracle/ into worktrees
A repo-root `.worktreeinclude` listing `.oracle/` was created so the git-ignored source is copied into any future worktree-isolated agent run.

**Rationale:** `.oracle/` is git-ignored, so it is absent from worktrees by default; oracle-dependent work run in a worktree would otherwise find no source. Not needed for this phase (wave-2 ran inline on the primary checkout) but a durable safeguard.
**Source:** this session (owner directive)

## Lessons

### Independent reviewers beat a single lenient pass on near-verbatim
The Chapter-3 reviewer accepted the canonical Intent sentences as "fact-like"; the Chapter-4 and Chapter-5 reviewers independently flagged the same construct as near-verbatim. The stricter verdict was the correct one for a public repo.

**Context:** running each family's review as a separate reviewer surfaced a systematic DST-04 exposure that one lenient reviewer had waved through -- the strict standard was then applied uniformly, including to already-committed leaves.
**Source:** 08.1-02..05-SUMMARY.md (oracle re-gates)

### Collecting Parameter's liability had to be assembled, not quoted
Kerievsky's Collecting Parameter has an empty explicit-liabilities list; its downside was assembled from the surrounding prose plus the Visitor contrast (tied to a shared interface, reads poorly across heterogeneous objects).

**Context:** the oracle-reviewer confirmed the assembly was faithful to what the prose supports and not fabricated -- documenting an implied downside is legitimate where the source implies but does not enumerate it (D-07).
**Source:** 08.1-06-SUMMARY.md, 08.1-06-PLAN.md (D-07)

### The org monthly spend limit can kill background oracle agents mid-run
Three background oracle-reviewer re-gates terminated with a spend-limit API error; they resumed cleanly from transcript once the limit reset (no work lost, chapter context preserved).

**Context:** quota-exceeded is a wait-and-resume condition, not a retry-now one; a resumed agent re-reads only the changed draft files, not the whole chapter.
**Source:** this session (background-task notifications)

### A stray shell `cd` produced a false ".oracle missing" alarm
An early `cd` into the phase subdirectory left the Bash cwd drifted, so `ls .oracle/` checked the wrong directory and reported the source folder absent -- when it was present at the repo root all along.

**Context:** reset the shell cwd to the repo root and prefer root-relative / absolute paths; do not conclude a git-ignored asset is missing from a single relative `ls`.
**Source:** this session

### tsconfig strictness shapes what an example can omit
The workspace tsconfig is `strict: true` but has no `noUnusedLocals` and no DOM lib, so examples are pure (no console/DOM) and may leave demo values unused; `noUncheckedIndexedAccess` is off, so idiomatic Iterator indexing needs no bounds guard.

**Context:** knowing the exact compiler flags up front let all 28 example fences compile first-try under the deterministic layer.
**Source:** .claude/skills/lz-refactor-workspace/tsconfig.json, 08.1-REVIEW.md

## Patterns

### Clean-room oracle loop
Author blind -> run the deterministic layer FIRST (tsc + hygiene + per-leaf contract) -> spend one oracle-reviewer call sub-batched ~6-8 leaves -> revise blind from directives -> iterate to convergence (~3-round cap) -> escalate non-convergence via AskUserQuestion.

**When to use:** authoring public content that must be faithful to a copyrighted source without the authoring context ever reading that source.
**Source:** 07-ORACLE-MODEL.md, 08.1-02..06-SUMMARY.md

### Deterministic layer before spending a reviewer call
Gate cheap machine checks (tsc --strict, ASCII/hygiene, per-leaf section rg) to green BEFORE invoking the (expensive, adversarial) oracle-reviewer.

**When to use:** any loop where a subagent review is the costly step -- never spend it on a draft that a free deterministic check would reject.
**Source:** 08-LEARNINGS.md (cited), 08.1-02..06-SUMMARY.md

### Pipelined inline authoring across file-disjoint plans
While plan N's oracle review runs in the background, author plan N+1's (file-disjoint) leaves; process each verdict as its notification lands.

**When to use:** inline multi-plan waves where the review step is background-able and the plans own disjoint files -- overlaps authoring wall-clock with review wall-clock.
**Source:** this session

### Deliberately re-domained examples to dodge too_close_to_source
Each pattern's `## Example` uses a domain distinct from the book's canonical illustration (e.g. Composite as a bill-of-materials, Flyweight as a forest of trees, Visitor as a staff-bonus walk).

**When to use:** authoring pattern/refactoring examples under a near-verbatim (DST-04) gate -- an independent domain is required, not optional.
**Source:** 08.1-RESEARCH.md (Pitfall 3), 08.1-03..05-SUMMARY.md

### Resume a chapter-loaded reviewer via SendMessage for cheap re-gates
For round-2+ gates, SendMessage the same oracle-reviewer agent (source chapter already in its context) instead of spawning a fresh one that re-reads the whole chapter.

**When to use:** iterative convergence loops against a large source chapter -- resuming reuses the loaded source and re-reads only the changed drafts.
**Source:** this session

## Surprises

### Reviewer strictness varied on the identical construct
The same canonical-Intent-sentence pattern drew `too_close_to_source: false` from one reviewer and `true` from two others in the same phase.

**Impact:** forced a catalog-wide intent-paraphrase pass (including re-opening the already-committed Creational leaves) and a `fix(08.1-02)` follow-up commit; converged all 28 leaves to a uniform strict standard.
**Source:** 08.1-02..05-SUMMARY.md

### The only review findings were in the non-shipped harness, not the shipped catalog
Code review returned 0 Critical / 2 Warning / 3 Info; both Warnings (a silently-dropped unterminated fence; two divergent slug functions) are dormant/latent issues in the non-shipped checker harness, and the shipped 28 leaves + 2 READMEs + SKILL.md were clean.

**Impact:** no shipping blocker; the harness findings are recorded as advisory in 08.1-REVIEW.md for future hardening rather than fixed in-phase (harness .mjs is exempt from the review gate).
**Source:** 08.1-REVIEW.md

### STATE.md plan counter went stale because wave 2 ran inline
Because the wave-2 content plans ran inline in the main context (D-13) rather than through gsd-executor, they never advanced STATE.md's plan counter, which sat at 2/7 until the Wave-3 executor set it directly to 7/7.

**Impact:** a blind `advance-plan` would have incremented the stale value incorrectly; the counter had to be set directly. Inline-execution phases need explicit tracking writes the executor path would otherwise handle.
**Source:** 08.1-07-SUMMARY.md
