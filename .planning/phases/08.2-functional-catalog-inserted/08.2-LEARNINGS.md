## Lessons

### The `cd <path> || <main-repo-fallback>` idiom is a recurring cwd-drift trap for worktree executors
FOUR separate executors this phase (wave-1, plan-05, plan-06, and the code-reviewer's verification run) hit a `cd ... || <fallback>` whose first arm resolved to the MAIN repo workspace, silently running checkers against the main tree and producing spurious FAILs. All self-recovered, no bad writes, but each lost time.

**Context:** The fix is to assert `git rev-parse --show-toplevel` ends in `.../worktrees/agent-<id>` immediately after any `cd`, and never write a main-repo fallback in the `||` arm. This warning must be pasted into every worktree-executor prompt.
**Source:** 08.2-01-SUMMARY.md, 08.2-05-SUMMARY.md, 08.2-06 execution, code-reviewer partial output

### A gate that validates only the FIRST match silently permits a wrong SECOND one (false-pass)
`check-functional` C4 validated only the first `Functional alternative:` line per OO leaf, so it did not enforce "exactly one". validate-phase proved the false-pass by appending a second valid-resolving-but-wrong link to builder.md (gate stayed GREEN), then hardened C4 with `matchAll` + `length === 1`.

**Context:** When a contract says "exactly one X", the checker must count occurrences, not short-circuit on the first. Adversarial mutation is what surfaced it; the passing battery alone did not.
**Source:** 08.2-REVIEW.md (WR-01), 08.2-VALIDATION.md

### GitHub anchor slugs drop `/` — filename and anchor can legitimately differ
`Replace One/Many Distinctions with Composite` has filename `replace-one-many-...md` but its GitHub heading anchor is `#replace-onemany-distinctions-with-composite` (githubSlug drops the slash). Cross-links must target the anchor form, not the filename form.

**Context:** Any N:1 row or `Functional alternative:` link pointing at a residual anchor derived from a slash-containing pattern name must use the slash-dropped slug. Recorded so Wave 4 got all 55 links right.
**Source:** 08.2-02-SUMMARY.md

### `structuredClone` is not in the `tsc --strict` lib baseline used by the harness
The structural-clone leaf could not put `structuredClone` inside a compiled fence (absent from `lib.es2021`); the compiled Example used a spread copy and named `structuredClone` in prose only.

**Context:** TS Examples that must compile under the pinned extractor are constrained to the configured lib; runtime-only globals go in prose, not fences.
**Source:** 08.2-03-SUMMARY.md

### A false-FAIL gate mismatch is an acceptable residual; a false-PASS is not
Review WR-02 (C1 `## Example` loose match vs sectionBody exact) fails toward RED (safe direction) so it was left as a documented residual; WR-01 (false-PASS) was fixed. The validate step triaged by failure direction, not by severity label.

**Context:** When hardening gates under time/token pressure, prioritize false-PASS paths (silent wrong-accept) over false-FAIL paths (noisy but safe).
**Source:** 08.2-VALIDATION.md

## Patterns

### Wave-based parallel worktree authoring of disjoint leaf sets
Wave 2 ran three executors in parallel isolated worktrees (6 + 7 + 6 = 19 leaves), each owning a disjoint file set, then merged back (ff-only the first, --no-ff the rest) with a single post-merge integration re-run.

**When to use:** Content-authoring phases where the unit of work partitions cleanly by file with no shared-file writes. Verify zero cross-branch file overlap before dispatch; run the integration checker (here check-crossrefs across all leaves) only AFTER merge, since each executor only validated its own subset.
**Source:** 08.2-02/03/04-SUMMARY.md, orchestration

### Instrument-first RED-to-GREEN convergence across waves
Wave 1 RED (0/19, 88 fails) -> Wave 2 partial (19/19 leaves present, 88 fails, reason shifted) -> Wave 3 (README+notes, 55 fails) -> Wave 4 (55 links, 0 fails / GREEN). Each wave drove a named subset of the gate's failures to zero.

**When to use:** Any phase where a single committed checker can express the full acceptance contract up front; the failing-count trajectory becomes the progress metric.
**Source:** all 08.2 SUMMARYs, battery logs

### Adversarial mutation probe to prove a checker discriminates
validate-phase confirmed each check is discriminating by transiently mutating shipped content (break a `Use when:`, a Correspondence enum, add a duplicate link), observing RED, then `git checkout`-restoring. A GREEN battery alone does not prove the gate would catch a regression.

**When to use:** Nyquist/coverage audits of a checker-as-test-suite; whenever "the tests pass" needs to mean "the tests would fail on a real defect".
**Source:** 08.2-VALIDATION.md

### N:1 leaf residual-anchor convention
N:1 idiom leaves carry per-pattern `### <Pattern>` subsections under "When each fits" that provide the residual anchors the README map rows target; 1:1 and native leaves use prose and are targeted at the bare leaf.

**When to use:** Any many-to-one index where one target file serves several source entries and each source needs a distinct deep link.
**Source:** 08.2-02-SUMMARY.md, skill-reviewer verdict

## Surprises

### The failing-check count stayed 88 from Wave 1 to Wave 2 despite 19 leaves appearing
Wave 1 RED = 88 (19 missing leaves + 55 missing links + 14 missing notes). After Wave 2 authored all 19 leaves, the count was STILL 88 — the 19 "missing leaf" failures converted 1:1 into 19 "missing README row" failures.

**Impact:** A flat total-failure count masked real progress; the meaningful signal was the per-reason breakdown (leaves 0->19 present), not the aggregate. Reading only the SUMMARY total would have looked like no movement.
**Source:** battery logs Wave 1 vs Wave 2, 08.2-02-SUMMARY.md

### Usage/session limits interrupted three agents mid-task; all resumed cleanly via SendMessage
The wave-1 executor, the code-reviewer, and the verifier were each terminated early by org spend / session limits, then resumed from transcript by SendMessage to their agentId with a ground-truth re-anchor message — no work lost, no re-execution.

**Impact:** Background-agent resume-from-transcript is robust for long GSD phases under quota pressure; capturing exact on-disk state (commits, worktree tips) before resuming let each agent continue rather than restart.
**Source:** orchestration, 08.2-01-SUMMARY.md

### The cwd-drift trap recurred even after being warned about it in prompts
Every worktree-executor prompt this phase included an explicit "never use `cd || <fallback>`; assert show-toplevel" instruction, yet the trap still fired four times (all self-recovered).

**Impact:** Prompt-level warnings alone did not prevent the drift; a structural guard (the executor's own path-safety assertion running BEFORE any checker command) is what caught it each time. Suggests the guard should be a hard first-action gate, not advisory prose.
**Source:** 08.2-05-SUMMARY.md, 08.2-06 execution
