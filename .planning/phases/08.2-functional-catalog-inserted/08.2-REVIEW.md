## Summary

Reviewed the phase 08.2 functional-catalog work with an adversarial, false-hypothesis-first stance,
concentrating effort where the risk actually lives: the 530-line `check-functional.mjs` gate, the
TypeScript `## Example` fences in all 19 idiom leaves, the N:1 pattern->idiom map in the README, and
the dual-mapping OO leaves (`mediator`, `iterator`, `builder`, `null-object`) whose `Functional
alternative:` link had a real chance of pointing at the wrong idiom.

I traced every one of the 19 leaf examples end to end for behavior preservation and confirmed each
Before->After produces identical observable output (Circle/Rectangle areas; the formatter pipe;
`checkout(percentageDiscount(0.1), 100) == 90`; `greet` returning `"unknown"`/name; the transducer
`evenSquares([1..4]) == [4,16]`; etc.). I also re-derived every anchor with the shared `githubSlug`
helper and cross-checked the C1-C8 gate conditions against the locked D-01..D-14 contract.

**No correctness or security defects were found in the shipped content.** All 23 GoF, 27 Kerievsky,
5 extra, and 15 Fowler-analog map rows resolve to a semantically correct idiom; the dual-mapped OO
leaves each forward-link to their *dissolution* idiom (Mediator->Reducer and Store, Iterator->Generator,
Builder->Function Composition, Null Object->Option and Either) while the `alternative-to` native FP
idioms reach back via their own `Correspondence` -- the design is internally consistent. The checker
is a read-only, node-builtins-only tool over local repo files, so there is no injection surface.

The 5 findings are all gate-hardening / doc-clarity issues (3 Warning, 2 Info): places where the
checker's stated contract is stronger than what the code enforces (so future content drift could slip
through), one latent false-fail from two checks disagreeing on the same heading, and one pedagogical
clarity issue in the `alternative-to` leaves' `mechanics:` lines. None block shipping.

## Warnings

### WR-01: C4 does not enforce "EXACTLY one `Functional alternative:` line" despite its own contract

**File:** `.claude/skills/lz-refactor-workspace/tools/check-functional.mjs:437`
**Issue:** The header contract (lines 17-22) states each OO leaf "carries EXACTLY one `Functional
alternative:` link". The implementation only matches the FIRST occurrence:
```js
const line = text.match(/^Functional alternative:\s*(.+?)\s*$/m);
```
A leaf with two `Functional alternative:` lines would pass C4 with only the first validated. If the
second line pointed to a valid-but-wrong idiom anchor, neither C4 (checks first only) nor
check-crossrefs (only checks that the link *resolves*, not that it is semantically correct) would flag
it. Today all 55 leaves are authored with exactly one line, so there is no current failure -- this is a
future-drift gate weakness where the documented "exactly one" cap is not actually enforced.
**Fix:** Assert the count, e.g.:
```js
const all = [...text.matchAll(/^Functional alternative:\s*(.+?)\s*$/gm)];
if (all.length !== 1) {
  report(false, `${dir}/${base}.md`, `expected exactly one Functional alternative: line, found ${all.length}`);
  continue;
}
const line = all[0];
```

### WR-02: C1 and `sectionBody` disagree on how to match the `## Example` heading (latent false-fail)

**File:** `.claude/skills/lz-refactor-workspace/tools/check-functional.mjs:328` vs `:148`/`:338`
**Issue:** The C1 template check accepts a loose Example heading:
```js
if (!/^##\s+Example\b/m.test(leaf.text)) { missing.push("## Example"); }   // line 328 -- \b allows trailing text
```
but `sectionBody` demands an exact heading:
```js
const re = new RegExp(`^##\\s+${escapeRe(heading)}\\s*$`, "m");            // line 149 -- requires "## Example" with nothing after
```
If a future leaf uses a descriptive heading like `## Example (Before -> After)`, C1 passes (`\b`
after "Example") but `sectionBody(leaf.text, "Example")` returns `null`, so `hasLiveTs` is `false`
and the leaf FAILS with the misleading message `non-ignore ts fence in ## Example` -- even though the
fence is present. The two checks should agree. (Note: C1's `## When each fits` check at line 332 and
`sectionBody` *do* agree, both exact -- only Example is inconsistent.) All 19 current leaves use a bare
`## Example`, so there is no live failure; this is a diagnostic-reliability defect that surfaces on
template drift.
**Fix:** Make C1 exact to match `sectionBody` (`/^##\s+Example\s*$/m`), or make `sectionBody` accept a
heading prefix. Prefer tightening C1 so the template stays canonical.

### WR-03: `alternative-to` leaves' `mechanics:` lines point to the OO construct the leaf argues against

**File:** `plugins/lz-tdd/skills/lz-refactor/references/functional-catalog/option-and-either.md:91`, `.../lens-and-optics.md:70`
**Issue:** The `dissolves-from` leaves give real transformation mechanics (`run X in reverse -- lift each
visit method into one arm of the switch`, etc.). The `alternative-to` leaves instead emit a bare
pointer with no steps and no "in reverse", and in two cases the linked refactoring produces the very OO
construct the leaf tells you NOT to build:
- `option-and-either.md:91` -- `mechanics: [Introduce Null Object] ...`. Introduce Null Object builds a
  Null Object; the example builds an `Option<User>`. The leaf's whole thesis is Option as the
  *alternative to* Null Object.
- `lens-and-optics.md:70` -- `mechanics: [Hide Delegate] ...`. Hide Delegate conceals structure behind
  delegating methods; the leaf explicitly says "a lens deliberately exposes the path instead."
Under the label `mechanics:` (which reads as "here is the recipe to reach the After"), pointing at the
contrasted refactoring is misleading. This is a pedagogical-content defect (the product here *is* the
guidance). `normalized-store` (-> Change Value to Reference) and `currying` (-> Extract Parameter) are
fine; `functor-and-monad` and `transducers` are borderline.
**Fix:** For `alternative-to` leaves, relabel the pointer (e.g. `related refactoring:` or `OO
alternative:`) so it is not read as the mechanics of producing the functional form, or point it at the
genuinely applicable step. If the current wording is the intended template, tighten D-01 to say so and
add a one-word direction cue.

## Info

### IN-01: C4's "55" and C1's "leaf files found" are informational, not asserted -- header overstates enforcement

**File:** `.claude/skills/lz-refactor-workspace/tools/check-functional.mjs:428-433`, `:522`
**Issue:** The header says C4 checks "EXACTLY the 55 (23 gof + 27 kerievsky + 5 extra)", but the code
only iterates whatever `collectOoLeafFiles` returns and prints `expected 55` as an `[INFO]` line
(line 522) without failing when the count differs. If an entire OO catalog dir went missing, C4 would
still pass (fewer leaves checked, all present ones valid). This is acceptable because the sibling gates
(`check-gof` = 23, `check-kerievsky` = 27, `check-extra-patterns` = 5) DO assert those counts, so
defense-in-depth covers it -- but the header wording implies this gate enforces the 55 itself, which it
does not.
**Fix:** Either assert `ooChecked === 55` (and each dir's contribution), or soften the header comment to
say the 55-count is enforced by the sibling checkers.

### IN-02: `collectLeaves` H1 detection does not strip code fences

**File:** `.claude/skills/lz-refactor-workspace/tools/check-functional.mjs:192`
**Issue:** H1 counting uses `lines.filter((l) => /^#\s+\S/.test(l))` over the raw file, not fence-aware.
A future leaf whose fenced example contains a column-0 `# ...` line (a shell/YAML/markdown-in-markdown
snippet) would be miscounted as a second level-1 heading and false-fail the "exactly one level-1
heading" check. Current leaves use only `ts` fences whose comments are `//`, so there is no impact
today, and this matches the established sibling-checker pattern -- noted for awareness only.
**Fix:** If ever needed, skip lines inside ``` fenced blocks when collecting H1s (same guard
check-crossrefs documents at its top).
